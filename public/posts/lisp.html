Lisp Q&A (Beginner to Advanced)

A comprehensive question-and-answer guide to Lisp — from the fundamentals to advanced topics. Use it to study, review, or prepare for interviews.

Beginner

Q: What is Lisp?

Lisp is a family of programming languages known for:

  • Prefix notation
  • Heavy use of lists
  • Code-as-data (homoiconicity)
  • Powerful macro systems

Popular Lisp dialects include Common Lisp, Scheme, Racket, and Clojure.

Q: Why is Lisp called a "list processor"?

Because Lisp programs and data are both represented primarily as lists. For example, code like (+ 1 2) is itself a list: an operator followed by arguments.

Q: What is an S-expression?

An S-expression (symbolic expression) is Lisp’s core syntax form:

  • Atoms (numbers, symbols, strings)
  • Lists (parenthesized expressions)

Examples:

  • Atom: 42, x, "hello"
  • List: (+ 1 2), (define x 10)

Q: What does prefix notation mean?

In Lisp, operators come before operands. 

(+ 1 2)     ; 3
(* 3 4)     ; 12
(- 10 7)    ; 3

This makes expression parsing simple and uniform.

Q: What are atoms and lists?

  • Atom: an indivisible value (number, symbol, string, etc.)
  • List: an ordered sequence enclosed in parentheses
42                 ; atom
foo                ; atom (symbol)
(1 2 3)            ; list
(+ 1 (* 2 3))      ; nested lists

Q: What is the REPL?

REPL = Read, Eval, Print, Loop. It is an interactive environment where you type expressions and immediately see results.

It is central to Lisp development and exploratory programming.

Q: What are symbols in Lisp?

Symbols are identifiers used for names and can also be used as data. 

(setq x 10)   ; x is a symbol bound to 10
'x            ; quoted symbol x (data, not variable lookup)

Q: What is quoting and why is it needed?

Normally, Lisp evaluates lists as function calls. Quote prevents evaluation and treats the expression as literal data. 

(+ 1 2)      ; evaluated => 3
'(+ 1 2)     ; literal list => (+ 1 2)
'(a b c)     ; list of symbols

Q: How do you define variables?

(Common Lisp style) 

(defparameter *name* "Alice")
(defvar *count* 0)
(let ((x 10)
      (y 20))
  (+ x y))   ; 30
  • defparameter: defines or redefines a global variable
  • defvar: defines only if not already bound
  • let: creates local bindings

Q: How do you define functions?

(defun greet (name)
  (format t "Hello, ~a!~%" name))

(defun add (a b)
  (+ a b))

Call with (greet "Alice") or (add 2 3).

Q: How do conditionals work in Lisp?

(if (> 5 3)
    "yes"
    "no")

(cond
  ((< x 0) "negative")
  ((= x 0) "zero")
  (t       "positive"))
  • if for two-way branching
  • cond for multi-branch logic
  • t means true/default branch

Q: What is NIL?

NIL represents both:

  • The empty list
  • Boolean false

Everything else is truthy in most Lisp dialects.

Q: How do lists work fundamentally?

(setq nums '(1 2 3 4))
(car nums)        ; 1       first element
(cdr nums)        ; (2 3 4) rest of list
(cons 0 nums)     ; (0 1 2 3 4)
(list 1 2 3)      ; (1 2 3)

Q: What is the difference between EQ, EQL, EQUAL, and EQUALP? (Common Lisp)

  • eq: identity (same object, mostly symbols)
  • eql: like eq, plus same-type numbers/chars with same value
  • equal: structural equality (lists/strings)
  • equalp: more permissive (case-insensitive strings, numeric coercion)

Use equal for most structural comparisons.

Intermediate

Q: What is recursion and why is it common in Lisp?

Recursion is defining a function in terms of itself. Because lists are naturally recursive (head + tail), recursion is idiomatic. 

(defun factorial (n)
  (if (<= n 1)
      1
      (* n (factorial (- n 1)))))

Q: How do map/filter/reduce patterns look in Lisp?

(mapcar #'1+ '(1 2 3))                 ; (2 3 4)
(remove-if-not #'evenp '(1 2 3 4 5))   ; (2 4)
(reduce #'+ '(1 2 3 4))                ; 10

These higher-order operations reduce explicit looping.

Q: What are lambda functions?

Lambda expressions define anonymous functions inline. 

(funcall (lambda (x) (* x x)) 5)       ; 25
(mapcar (lambda (x) (* 2 x)) '(1 2 3)) ; (2 4 6)

Q: What is lexical vs dynamic scope?

  • Lexical scope: variable visibility depends on code structure
  • Dynamic scope: visibility depends on runtime call chain

Modern Lisp code usually prefers lexical scope. Common Lisp also supports dynamic scope via special variables.

Q: What are special variables in Common Lisp?

Special variables are dynamically scoped variables, often named with *earmuffs*. 

(defparameter *debug* t)

(defun log-msg (msg)
  (when *debug*
    (format t "[DEBUG] ~a~%" msg)))

They can be temporarily rebound with let.

Q: What are property lists and association lists?

;; Property list (plist)
(setq person '(:name "Alice" :age 30))
(getf person :name) ; "Alice"

;; Association list (alist)
(setq phonebook '(("Alice" . "1234") ("Bob" . "5678")))
(cdr (assoc "Bob" phonebook :test #'string=)) ; "5678"

Q: How do you handle iteration besides recursion?

(loop for x in '(1 2 3 4)
      collect (* x x)) ; (1 4 9 16)

(dolist (x '(a b c))
  (print x))

(dotimes (i 3)
  (print i))

Common Lisp’s loop is highly expressive.

Q: What are multiple return values?

Functions can return multiple values efficiently. 

(defun divmod (a b)
  (values (floor a b) (mod a b)))

(multiple-value-bind (q r) (divmod 17 5)
  (format t "q=~a r=~a~%" q r))

Q: What are reader macros?

Reader macros extend syntax at read time. Example: quote shorthand. 

'(1 2 3)   ; reader expands to (quote (1 2 3))

In Common Lisp, the reader is customizable for DSL-like syntax.

Q: What is quasiquote / unquote?

Quasiquote helps build list templates. 

(let ((x 10))
  `(1 2 ,x 4))     ; (1 2 10 4)

(let ((xs '(3 4)))
  `(1 2 ,@xs 5))   ; (1 2 3 4 5)
  • ` quasiquote
  • , unquote a single value
  • ,@ splice list contents

Q: How does error handling work?

(handler-case
    (/ 1 0)
  (division-by-zero (c)
    (format t "Caught error: ~a~%" c)))

Common Lisp has a rich condition system (more powerful than simple exceptions).

Q: What is garbage collection in Lisp?

Most Lisp implementations use garbage collection. You allocate objects freely; unreachable objects are reclaimed automatically.

This encourages high-level, allocation-friendly programming styles.

Q: What are vectors and hash tables, and when should you use them?

Lists are great for recursive processing and symbolic code, but not always best for random access. 

;; Vector
(setq v #(10 20 30))
(aref v 1) ; 20

;; Hash table
(setq ht (make-hash-table :test #'equal))
(setf (gethash "name" ht) "Alice")
(gethash "name" ht) ; "Alice", T
  • Use lists for sequential/symbolic data
  • Use vectors for indexed access
  • Use hash tables for key-value lookups

Advanced

Q: What are macros and why are they powerful?

Macros transform code before evaluation. Unlike functions, macros receive unevaluated code forms and return new code. 

(defmacro unless (condition &body body)
  `(if (not ,condition)
       (progn ,@body)))

This enables creation of new language constructs.

Q: What is homoiconicity?

Homoiconicity means Lisp code uses the same structure as Lisp data (lists/symbols). So programs can manipulate programs naturally.

This is the foundation of macro systems and metaprogramming.

Q: How do macros differ from functions?

  • Function arguments are evaluated before the call
  • Macro arguments are raw syntax forms
  • Macro expansion produces code that is evaluated later

Use macros for syntactic abstraction and control-flow constructs, not ordinary computation.

Q: How do you avoid common macro pitfalls?

  • Avoid variable capture (use gensym)
  • Understand evaluation timing
  • Expand and inspect with macroexpand-1
  • Prefer functions unless syntax transformation is truly needed
(defmacro with-temp ((var value) &body body)
  (let ((g (gensym "TMP-")))
    `(let ((,g ,value))
       (let ((,var ,g))
         ,@body))))

Q: What is tail recursion and TCO?

Tail recursion means the recursive call is in final position. Tail-call optimization (TCO) can reuse stack frames.

Scheme requires proper tail calls. Common Lisp does not guarantee TCO (implementation-dependent).

Q: What is the Common Lisp Object System (CLOS)?

CLOS is Common Lisp’s dynamic object system with:

  • Generic functions
  • Multiple dispatch
  • Method combinations
  • Runtime class redefinition
(defclass animal ()
  ((name :initarg :name :accessor name)))

(defgeneric speak (obj))

(defmethod speak ((a animal))
  (format t "~a makes a sound.~%" (name a)))

Q: What is multiple dispatch?

Method selection can depend on the types of multiple arguments (not just one receiver). 

(defgeneric collide (a b))

(defmethod collide ((a asteroid) (b ship)) ...)
(defmethod collide ((a ship) (b asteroid)) ...)

This models interactions elegantly.

Q: What is the Lisp condition/restart system?

Beyond exceptions, Lisp allows signaling conditions and offering restarts (recovery strategies) without immediately unwinding the stack.

This supports interactive and resilient error recovery in long-running systems.

Q: How do packages and namespaces work in Common Lisp?

Packages manage symbol visibility and prevent name clashes. 

(defpackage :my-app
  (:use :cl)
  (:export :main))

(in-package :my-app)

(defun main ()
  (format t "Hello from package MY-APP~%"))

Q: What is metaprogramming in Lisp?

Metaprogramming means writing code that generates or transforms code:

  • Macros
  • Code walkers
  • Reader extensions
  • Runtime code generation

Lisp is especially strong at building DSLs and reusable abstractions.

Q: How can you build a small DSL with macros?

A common pattern:

  • Represent DSL forms as lists
  • Use macros to translate forms into plain Lisp
  • Keep runtime simple; move complexity to macro-expansion time

This gives expressive APIs with little runtime overhead.

Q: How do compilation and loading work in Common Lisp?

Typical workflow:

  • Write source files
  • Compile with compile-file
  • Load FASL output with load
(compile-file "my-app.lisp")
(load "my-app.fasl")

Compiled code is usually faster and can improve startup behavior.

Q: What is continuation support in Lisp dialects?

Some dialects (especially Scheme implementations) support first-class continuations (call/cc), enabling advanced control flow:

  • Backtracking
  • Coroutines
  • Non-local exits
  • User-level schedulers

Common Lisp usually uses different control abstractions.

Q: What are performance considerations in Lisp?

  • Add type declarations in hot paths (implementation-dependent gains)
  • Avoid unnecessary generic dispatch in tight loops
  • Profile before optimizing
  • Use compiler warnings and optimization settings
  • Choose data structures carefully (lists vs vectors vs hash tables)

Q: How do you debug Lisp programs effectively?

  • Use REPL-driven development
  • Inspect macro expansions (macroexpand-1)
  • Trace functions (trace, untrace)
  • Use implementation debugger/inspector
  • Keep functions small and test interactively

Q: What are Common Lisp vs Scheme vs Clojure differences (high level)?

  • Common Lisp: broad standard, CLOS, powerful macros, dynamic development model
  • Scheme: minimal core, elegant semantics, proper tail calls
  • Clojure: modern Lisp on JVM/CLR, immutable-by-default data, strong concurrency primitives

Choose based on ecosystem, runtime needs, and project goals.

Q: What are common beginner mistakes in Lisp?

  • Forgetting quote when data should not be evaluated
  • Confusing symbols with string values
  • Overusing macros where functions are better
  • Ignoring parentheses structure (especially nested forms)
  • Not leveraging the REPL for incremental development

A good habit: evaluate and test expressions in small pieces.
