Lisp Interpreter Internals

Q: How does Emacs evaluate Lisp code?

Evaluation starts in the C function: 

Feval()

Located in: 

src/eval.c

The evaluator:

  1. Reads Lisp object
  2. Determines object type
  3. Resolves symbol
  4. Evaluates function
  5. Returns Lisp object

Evaluation pipeline: 

Reader → Parser → Evaluator → Return Value

Q: What is a Lisp Object internally?

Everything in Emacs is represented as: 

Lisp_Object

Defined in: 

src/lisp.h

Stores:

  • integers
  • symbols
  • strings
  • cons cells
  • vectors
  • buffers
  • windows
  • processes

Internally tagged with type information.

Q: How are symbols stored internally?

A symbol contains:

  • value cell
  • function cell
  • property list
  • symbol name

Internally: 

struct Lisp_Symbol

Example: 

(setq x 10)

Modifies symbol value cell.

Q: What happens when a function is called?

Example: 

(+ 1 2)

Internal process:

  1. resolve symbol +
  2. locate function cell
  3. evaluate arguments
  4. dispatch builtin or bytecode
  5. return result

C entry point: 

funcall_subr()

Memory Management

Q: How does garbage collection work?

Emacs uses mark-and-sweep garbage collection.

Phases:

  1. mark reachable objects
  2. sweep unused objects
  3. reclaim memory

Controlled by: 

gc-cons-threshold

Q: What triggers garbage collection?

GC occurs when allocated memory exceeds threshold.

Check statistics: 

(garbage-collect)

Returns allocation information.

Q: Why does excessive consing slow Emacs?

Consing creates temporary objects.

Example bad pattern: 

(setq x (append x '(1)))

Better: 

(push 1 x)

Excessive consing causes frequent GC cycles.

Bytecode Virtual Machine

Q: How does byte compilation work?

Compiler: 

(byte-compile-file "file.el")

Produces: 

file.elc

Compiler stages:

  1. parse forms
  2. generate bytecode
  3. optimize constants
  4. write bytecode file

Q: What is inside .elc files?

Contains:

  • constants
  • bytecode instructions
  • function definitions

Executed by bytecode VM.

Q: How does bytecode execution work?

Bytecode executed by: 

exec_byte_code()

Located in: 

src/bytecode.c

VM executes stack-based instructions.

Example instructions:

  • constant
  • call
  • jump
  • return
  • branch

Q: What is native compilation?

Modern Emacs supports native compilation.

Produces: 

.eln

Pipeline: 

Elisp → Bytecode → GCC JIT → Native Machine Code

Benefits:

  • faster execution
  • lower interpreter overhead

Buffer Internals

Q: What is a buffer internally?

Internally: 

struct buffer

Located: 

src/buffer.h

Stores:

  • text data
  • markers
  • local variables
  • syntax table
  • undo history
  • point position

Q: How is buffer text stored?

Emacs uses a gap buffer.

Structure: 

TEXT | GAP | TEXT

Insertion occurs at gap.

Advantages:

  • fast insertion near cursor
  • low memory movement

Q: Why does moving point matter?

Moving point may require moving gap.

Large gap movement:

  • copies memory
  • slows editing

This affects performance.

Redisplay Engine

Q: What is the redisplay engine?

Responsible for drawing editor contents.

Main C file: 

src/xdisp.c

Handles:

  • cursor rendering
  • line wrapping
  • syntax highlighting
  • overlays
  • text properties

Q: Why is redisplay expensive?

Redisplay recalculates:

  • visible lines
  • faces
  • overlays
  • line numbers
  • truncation

Excessive redisplay causes lag.

Q: How do packages accidentally slow redisplay?

Common causes:

  • too many overlays
  • expensive hooks
  • font-lock complexity
  • frequent buffer modifications

Window System

Q: Difference between frame and window internally?

Frame:

GUI window.

Window:

Viewport showing buffer.

Internal structs: 

struct frame
struct window

Q: How does window splitting work?

Window tree created internally.

Each node stores:

  • parent
  • child

Functions: 

(split-window)
(delete-window)

Minibuffer Architecture

Q: How does minibuffer work internally?

Minibuffer is a normal buffer.

Used for:

  • M-x
  • file completion
  • prompts
  • interactive commands

Example: 

(read-string "Name: ")

Q: How does completion work?

Completion pipeline: 

Input → Completion Table → Candidate Filtering → UI Renderer

Core functions: 

completing-read
try-completion
all-completions

Packages extending completion:

  • Vertico
  • Consult
  • Corfu

Font Lock Architecture

Q: How does syntax highlighting work?

Implemented using: 

font-lock-mode

Pipeline:

  1. regex search
  2. match patterns
  3. apply text properties
  4. redisplay

Q: Why can syntax highlighting become slow?

Reasons:

  • expensive regex
  • large files
  • repeated scans
  • too many overlays

Package System Internals

Q: How does require work?

Example: 

(require 'json)

Process:

  1. search load-path
  2. locate file
  3. evaluate file
  4. call provide
  5. register loaded feature

Q: What is load-path?

List of directories searched for packages.

Inspect: 

load-path

Q: What does provide do internally?

Example: 

(provide 'mypackage)

Registers feature in: 

features

Process System

Q: How does make-process work?

Creates asynchronous process. 

(make-process ...)

Internally:

  • spawn process
  • connect pipes
  • register process object

Q: What are process filters?

Process filters handle output. 

(make-process
 :filter ...)

Called when new process output arrives.

Q: What are process sentinels?

Sentinel handles state changes. 

(make-process
 :sentinel ...)

Runs when process:

  • exits
  • fails
  • stops

Hook System

Q: How do hooks work internally?

Hooks store function lists.

Example: 

(add-hook ...)

Internally: 

hook-variable → list of functions

Execution:

Functions called sequentially.

Q: Why can hooks cause performance issues?

Bad hooks:

  • expensive computations
  • repeated execution
  • synchronous I/O

Example: 

(after-change-functions)

Runs very frequently.

Package Performance

Q: Why do badly written packages slow Emacs?

Common mistakes:

  • repeated regex scans
  • too many overlays
  • synchronous shell-command
  • excessive consing
  • unnecessary redisplay
  • blocking network calls

Q: How do you profile Emacs?

Use built-in profiler. 

(profiler-start 'cpu)
(profiler-report)

Benchmark: 

(benchmark-run ...)
