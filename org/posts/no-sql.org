* NoSQL Q&A (Beginner - Advanced)

* Fundamentals

** Q1. What is NoSQL?
NoSQL (*Not Only SQL*) is a family of *non-relational* databases built for:
- distributed systems
- horizontal scaling
- flexible data structures

Unlike relational databases, NoSQL systems often trade some strict guarantees
(e.g., joins, fixed schema, strict consistency everywhere) for scalability and availability.

** Q2. What are the main categories of NoSQL databases?
- *Document stores*: MongoDB, Couchbase
- *Key-Value stores*: Redis, DynamoDB, Riak
- *Wide-Column stores*: Cassandra, HBase, ScyllaDB
- *Graph databases*: Neo4j, JanusGraph, Amazon Neptune

Sometimes grouped nearby (not always classic NoSQL):
- Search engines (Elasticsearch)
- Time-series DBs (InfluxDB)

** Q3. How does NoSQL differ from relational (SQL) databases?
*Relational (SQL):*
- fixed schema
- tables/rows
- joins
- strong ACID focus

*NoSQL (typical):*
- flexible schema
- multiple data models
- horizontal scale-first design
- denormalization over heavy joins
- often eventual consistency (or tunable consistency)

** Q4. When should you choose NoSQL over relational?
Choose NoSQL when you need:
- very high scale (especially horizontal)
- high write throughput
- evolving/flexible schema
- low-latency key-based access
- specialized models (graph, time-series, etc.)

Prefer relational when you need:
- complex joins
- strict multi-entity transactional integrity
- strong relational constraints

** Q5. What is a document database?
A document DB stores each record as a self-contained document (usually JSON/BSON),
often with nested objects and arrays.

Key point: documents in one collection do *not* need identical structure.

** Q6. What is a key-value store?
A key-value store is the simplest model:
- unique key -> opaque value

Strength:
- extremely fast reads/writes by key

Trade-off:
- limited querying beyond key lookup

** Q7. What is a wide-column (column-family) store?
A wide-column database stores rows by key, where rows can contain many columns,
often grouped into column families.

Best for:
- huge datasets
- write-heavy workloads
- predictable query patterns

** Q8. What is a graph database?
Graph DBs model:
- *nodes* (entities)
- *edges* (relationships)
- *properties* on both

They are optimized for relationship traversal (multi-hop queries), such as:
- social networks
- recommendation engines
- fraud detection

** Q9. Schema-on-read vs schema-on-write
- *Schema-on-write*: structure is enforced when data is written (common in SQL)
- *Schema-on-read*: data is stored flexibly, structure interpreted at read time (common in NoSQL)

Schema-on-read gives flexibility but moves more responsibility to the application.

** Q10. What is denormalization and why common in NoSQL?
Denormalization means duplicating related data so one query can return everything needed.

Why common:
- avoids expensive joins
- improves read speed and scalability

Cost:
- more work to keep duplicated data consistent

** Q11. What is the CAP theorem?
In distributed systems, during a network partition you usually choose between:
- *Consistency (C)* and
- *Availability (A)*

Partition tolerance (P) is required in real distributed systems, so trade-off is usually CP vs AP under partition.

** Q12. Examples of CP and AP systems
- *CP-leaning*: HBase, ZooKeeper, MongoDB (common majority settings), single-node Redis
- *AP-leaning*: Cassandra, DynamoDB-style systems, Riak, CouchDB

Note: many modern systems allow tuning between stronger consistency and higher availability.

** Q13. What is BASE?
BASE is an alternative mindset to strict ACID in distributed systems:
- *Basically Available*
- *Soft state*
- *Eventual consistency*

It prioritizes availability and accepts temporary inconsistency.

** Q14. What is eventual consistency?
If no new updates happen, all replicas eventually converge to the same value.
Implication: some reads may be temporarily stale.

** Q15. What is strong consistency?
A read always returns the latest committed write.
This usually requires coordination between replicas, which may increase latency or reduce availability during failures.

** Q16. Horizontal vs vertical scaling
- *Vertical scaling*: add CPU/RAM to one machine
- *Horizontal scaling*: add more machines and distribute data/load

NoSQL systems are usually designed for horizontal scaling via sharding + replication.

** Q17. What is sharding (partitioning)?
Sharding splits data across nodes using a shard/partition key, so each node stores only a subset.

Benefits:
- scale out capacity
- parallel processing
- distribute traffic

** Q18. What is replication?
Replication keeps multiple copies of data on different nodes.

Benefits:
- fault tolerance
- higher availability
- read scaling

Replication can be synchronous or asynchronous.

** Q19. Sharding vs replication
- *Sharding*: different chunks of data on different nodes (capacity/scale)
- *Replication*: same data copied to multiple nodes (availability/durability)

Most real systems use both together.

** Q20. Do NoSQL databases support transactions?
Yes, increasingly:
- most support atomic single-document/single-key operations
- many now support multi-document/multi-key ACID transactions

Trade-off: stronger transactional guarantees often reduce throughput or increase latency.

** Q21. What is a collection / table / keyspace?
These are logical groupings of records:
- *Collection* (MongoDB)
- *Table* (DynamoDB, Cassandra)
- *Keyspace* (Cassandra namespace for tables)
- *Database* (MongoDB namespace for collections)
  
** Q22. What is a primary key in NoSQL?
A primary key uniquely identifies a record.

Examples:
- key-value/document: usually one key
- wide-column: often composite (partition key + clustering/sort key)

** Q23. Partition key vs sort/clustering key
- *Partition key*: decides where data is stored (which partition/node)
- *Sort/Clustering key*: orders data within that partition

This design enables efficient range queries inside a partition.

** Q24. What is TTL (time to live)?
TTL sets automatic expiry for a record.

Common use cases:
- cache entries
- sessions
- temporary/ephemeral data

Used in systems like Redis, Cassandra, and DynamoDB.

** Q25. What is BSON?
BSON = Binary JSON (used by MongoDB).

It extends JSON with extra native types (e.g., ObjectId, Date, Decimal128, Binary) and efficient binary encoding.

** Q26. What is ObjectId in MongoDB?
ObjectId is MongoDB’s default value for =_id=:
- 12-byte unique identifier
- includes timestamp + machine/process info + counter

Useful property: roughly sortable by creation time.

** Q27. What is polyglot persistence?
Using multiple database technologies in one system, each for what it does best.

Example:
- Redis for caching
- MongoDB for documents
- Neo4j for relationship-heavy queries

** Q28. What is a secondary index?
An index on non-primary-key fields, enabling faster queries by those fields.

Trade-off:
- extra storage
- write overhead (index maintenance)

** Q29. What are NoSQL trade-offs?
*Pros:*
- scalability
- flexibility
- performance
- availability

*Cons:*
- often weaker default consistency
- fewer join/ad-hoc query capabilities
- denormalization complexity
- data duplication sync challenges
- requires up-front access-pattern design

** Q30. Is NoSQL really schemaless?
Not truly “no schema” — more like *flexible schema*.

The DB may not enforce strict structure, but applications still depend on an implicit schema.
Many NoSQL systems now support optional schema validation.

* Data Modeling (Clear Learning Version)

** Q31. How NoSQL modeling differs from relational modeling
Relational modeling starts with normalized entities + relationships.

NoSQL modeling starts with *access patterns* (queries first), then shapes data for fast reads/writes, often with denormalization.

** Q32. What is query-driven data modeling?
Design data layout based on how the app reads/writes data in real usage.

Process:
1) list key queries
2) design records so each query is efficient (ideally one operation)

** Q33. Embedding vs referencing in document DBs
- *Embedding*: nested subdocuments inside parent
  - fewer reads
  - atomic parent-level updates
- *Referencing*: separate documents linked by id
  - more flexible
  - may require extra queries/joins in app layer

** Q34. When embed vs reference?
*Embed when:*
- one-to-few
- always read together
- bounded size

*Reference when:*
- many-to-many
- unbounded growth
- shared data updated often
- child entities queried independently

** Q35. MongoDB document size limit
MongoDB max document size is *16 MB* BSON.

For larger file/binary content:
- use GridFS
- or object storage (S3, etc.)

** Q36. One-to-many modeling in document stores
- Few children -> embed array in parent
- Many children -> child docs with parent id reference
- Huge/unbounded -> separate collection + pagination

** Q37. Many-to-many modeling in NoSQL
Common options:
- arrays of references on both sides
- dedicated join/edge collection
- duplication per query pattern

Choose based on dominant read/write paths.

** Q38. Outlier pattern
Most records follow normal shape; rare large records are split/handled differently.
Goal: keep common-case documents small and fast.

** Q39. Bucket pattern
Group many small events into time buckets (e.g., per minute/hour/day document).

Benefits:
- fewer documents
- lower index overhead
- better ingestion efficiency

** Q40. Subset pattern
Store only a small hot subset (e.g., latest 10 items) in main document.
Store full history separately.

Benefit: faster hot reads with controlled document size.

** Q41. Computed pattern
Precompute values (count/sum/avg) on write instead of calculating on every read.

Trade-off:
- more complex writes
- much faster reads

** Q42. Extended reference pattern
Copy a few frequently used fields from referenced doc into main doc (e.g., customer name on order).

Benefit: avoid extra lookups for common views.

** Q43. Schema versioning pattern
Add =schema_version= field so app can read multiple document versions safely during migrations.

** Q44. Schema migration in schemaless DB
Common strategies:
- lazy migration (on read/write)
- background batch migration
- dual-write during transition

Best practice: use version fields and backward-compatible readers.

** Q45. Composite/compound key
A key composed of multiple parts (e.g., partition key + sort key).

Used to:
- colocate related records
- support range queries inside partition

** Q46. Choosing a good partition key
Good partition key should:
- have high cardinality
- spread traffic evenly
- still align with common queries

Avoid skewed keys that overload one partition.

** Q47. What is a hot partition / hot key?
A key/partition with disproportionate traffic, causing bottleneck and throttling on one node.

** Q48. How to mitigate hot partitions
- write sharding (random suffix)
- better key design (higher cardinality)
- time bucketing
- cache hot reads

** Q49. Why duplication is acceptable in NoSQL
NoSQL often trades storage for latency:
duplicated data can serve reads in one fast operation.

Trade-off remains: consistency management across copies.

** Q50. Keeping duplicated data consistent
Approaches:
- update all copies in app logic
- CDC/stream-based propagation
- reconciliation/background repair jobs
- accept eventual consistency where business allows it

** Q51. What is single-table design in DynamoDB?
Store multiple entity types in one table using generic PK/SK patterns and overloaded attributes.

Benefits:
- fewer network calls
- related data colocated
- cost/performance efficiency

Complexity:
- harder modeling and evolution

** Q52. Single-table design trade-offs
*Pros:* efficient access, fewer joins/requests, cost savings  
*Cons:* steep learning curve, key design complexity, limited ad-hoc analytics flexibility

** Q53. Tree/hierarchy: graph DB vs document DB
- *Graph DB*: parent/child edges, natural traversal
- *Document DB*: ancestors array, materialized path, or parent references

Pick based on read/write and traversal depth needs.

** Q54. Materialized path pattern
Store full hierarchy path as string (e.g., =/root/a/b=).

Benefit: subtree queries via prefix match  
Cost: moving a node requires updating descendant paths

** Q55. Time-series modeling in NoSQL
Typical approach:
- bucket by time window
- use time in sort/clustering key
- use TTL for retention
- pre-aggregate for dashboards

If time-series is core workload, consider dedicated TSDBs too.

* Document Databases (MongoDB-focused, Clear Learning Version)
** Q56. Insert documents
#+begin_src js
db.users.insertOne({ name: "Ada", age: 36, roles: ["admin"] });
db.users.insertMany([{ name: "Alan" }, { name: "Grace" }]);
#+end_src

** Q57. Query documents
#+begin_src js
db.users.find({ age: { $gte: 30 } });
db.users.findOne({ name: "Ada" });
#+end_src

** Q58. Common query operators
- Comparison: =$eq=, =$ne=, =$gt=, =$gte=, =$lt=, =$lte=, =$in=, =$nin=
- Logical: =$and=, =$or=, =$not=, =$nor=
- Element: =$exists=, =$type=
- Array: =$all=, =$elemMatch=, =$size=

** Q59. Update documents
#+begin_src js
db.users.updateOne({ name: "Ada" }, { $set: { age: 37 } });
db.users.updateMany({ active: false }, { $set: { archived: true } });
#+end_src

** Q60. Common update operators
=$set=, =$unset=, =$inc=, =$mul=, =$rename=, =$min=, =$max=, =$push=, =$pull=, =$addToSet=, =$pop=, =$currentDate=

** Q61. Delete documents
#+begin_src js
db.users.deleteOne({ name: "Ada" });
db.users.deleteMany({ archived: true });
#+end_src

** Q62. What is upsert?
Update matching document; if not found, insert new document.

#+begin_src js
db.counters.updateOne(
  { _id: "page" },
  { $inc: { views: 1 } },
  { upsert: true }
);
#+end_src

** Q63. Query nested fields
Use dot notation:

#+begin_src js
db.users.find({ "address.city": "Sofia" });
#+end_src

** Q64. Query arrays
#+begin_src js
db.posts.find({ tags: "mongodb" });
db.posts.find({ tags: { $all: ["db", "nosql"] } });
db.orders.find({ items: { $elemMatch: { qty: { $gt: 5 } } } });
#+end_src

** Q65. Projection (select fields)
#+begin_src js
db.users.find({}, { name: 1, email: 1, _id: 0 });
#+end_src

** Q66. What is aggregation pipeline?
A stage-based data processing pipeline:
=$match= -> =$group= -> =$project= -> =$sort= -> etc.

Used for analytics, reshaping, joins-like flows, and computed outputs.

** Q67. Aggregation example
#+begin_src js
db.orders.aggregate([
  { $match: { status: "paid" } },
  { $group: { _id: "$customerId", total: { $sum: "$amount" } } },
  { $sort: { total: -1 } },
  { $limit: 10 }
]);
#+end_src

** Q68. What does =$lookup= do?
Left outer join with another collection.

#+begin_src js
db.orders.aggregate([
  {
    $lookup: {
      from: "customers",
      localField: "custId",
      foreignField: "_id",
      as: "customer"
    }
  }
]);
#+end_src

** Q69. Why place =$match= early?
Early filtering reduces pipeline workload and can improve index usage.

** Q70. What does =$unwind= do?
Expands each array element into a separate document in the pipeline.
Useful before grouping/filtering by array members.

** Q71. =find()= vs =aggregate()=
- =find()=: simple filtering + projection
- =aggregate()=: multi-stage transforms, grouping, joins, computed fields

** Q72. Creating indexes
#+begin_src js
db.users.createIndex({ email: 1 }, { unique: true });
db.posts.createIndex({ author: 1, createdAt: -1 });
#+end_src

** Q73. Compound index and field order
Field order matters due to prefix rule:
index ={a:1, b:1}= supports queries on:
- =a=
- =a + b=
but not efficiently on =b= alone.

** Q74. Covered query
A query answered entirely by index (no document fetch).
Requirement: filter + returned fields all available in the index.

** Q75. Analyze query performance
Use:
#+begin_src js
db.users.find({ email: "a@b.com" }).explain("executionStats");
#+end_src

Check:
- =IXSCAN= (good index usage)
- =COLLSCAN= (full scan warning)
- docs examined vs docs returned

** Q76. Multikey indexes
Indexes on array fields.
MongoDB indexes each array element, enabling efficient array-content queries.

** Q77. TTL index
Auto-removes expired docs based on date field + TTL setting.

#+begin_src js
db.sessions.createIndex({ createdAt: 1 }, { expireAfterSeconds: 3600 });
#+end_src

** Q78. Text index and full-text search
#+begin_src js
db.articles.createIndex({ body: "text" });
db.articles.find({ $text: { $search: "nosql database" } });
#+end_src

For advanced ranking/features, use Atlas Search or Elasticsearch/OpenSearch.

** Q79. Partial vs sparse index
- *Sparse index*: only docs that contain indexed field
- *Partial index*: only docs matching a filter expression

Both reduce index size for targeted workloads.

** Q80. MongoDB transactions
- Single-document operations are atomic by default.
- Multi-document ACID transactions supported (replica sets/sharded), but with overhead.

Design preference: keep operations single-document atomic when possible.

** Q81. Read concern vs write concern
- *Write concern* (=w=, =j=): durability/acknowledgment level for writes
- *Read concern* (=local=, =majority=, =linearizable=): consistency level for reads

** Q82. Read preference
Controls where reads go:
=primary=, =secondary=, =primaryPreferred=, etc.

Trade-off:
- primary reads -> fresher
- secondary reads -> scale/locality, may be stale

** Q83. WiredTiger
MongoDB default storage engine with:
- document-level concurrency
- compression
- MVCC snapshots for consistent reads

** Q84. GridFS
Stores files larger than 16 MB by splitting into chunks in:
- =fs.files=
- =fs.chunks=

** Q85. MongoDB sharding (high level)
- shard key decides data distribution
- =mongos= routes requests
- config servers store metadata
- balancer migrates chunks for even distribution

** Q86. Change streams
Real-time stream of inserts/updates/deletes built from oplog.

Used for:
- event-driven apps
- cache invalidation
- projections/sync pipelines

* Key-Value & Caching (Redis-focused, Clear Learning Version)
** Q87. What is Redis?
Redis is an in-memory key-value store with rich data structures.

Common uses:
- caching
- sessions
- pub/sub
- queues/streams
- fast counters/leaderboards

** Q88. Redis data structures
Core: Strings, Lists, Sets, Sorted Sets, Hashes  
Also: Bitmaps, HyperLogLog, Streams, Geospatial  
Via modules: JSON, search, time-series, bloom filters

** Q89. Basic Redis string commands
#+begin_src text
SET key value
GET key
INCR counter
SETEX key 60 value
MSET k1 v1 k2 v2
#+end_src

** Q90. Redis expiration (TTL)
Set TTL with:
- =EXPIRE key seconds=
- or =SET key value EX seconds=

Expiration happens via lazy checks + active background sampling.
Use =TTL key= to see remaining lifetime.

** Q91. Redis eviction policies
When memory limit is reached, eviction depends on =maxmemory-policy=:
=noeviction=, =allkeys-lru=, =allkeys-lfu=, =volatile-lru=, =volatile-lfu=,
=volatile-ttl=, =allkeys-random=, =volatile-random=

** Q92. LRU vs LFU eviction
- *LRU*: evict least recently used keys
- *LFU*: evict least frequently used keys

LFU often works better for long-term hot keys.

** Q93. Redis persistence
- *RDB*: snapshots (compact, fast startup, potential recent data loss)
- *AOF*: append each write (more durable, larger files)
- Can combine both for balanced durability + recovery.
  
** Q94. Redis Sorted Set (ZSet) + use case
Sorted Set = unique members + numeric score (kept ordered).

Use cases:
- leaderboards
- priority queues
- ranked feeds
- rate limiting

#+begin_src text
ZADD leaderboard 100 "ada"
ZREVRANGE leaderboard 0 9 WITHSCORES
#+end_src

** Q95. Redis Pub/Sub
Publishers send messages to channels; subscribers receive in real time.

Important:
- not durable
- messages are lost if subscriber is offline

Use Streams for durability.

** Q96. Redis Streams
Append-only log for durable messaging:
- ordered IDs
- consumer groups
- acknowledgments

Useful for event pipelines with at-least-once processing.

** Q97. Caching patterns with Redis
- *Cache-aside*: app loads DB on miss, then fills cache
- *Write-through*: write cache + DB synchronously
- *Write-behind*: write cache first, DB async later
- *Read-through*: cache component fetches source on miss

** Q98. Thundering herd / cache stampede
When a hot key expires, many requests miss simultaneously and hammer DB.

Mitigations:
- request coalescing / single-flight lock
- jittered (staggered) TTLs
- early refresh
- background warmup/refresh

** Q99. Distributed lock with Redis
Acquire lock:
#+begin_src text
SET lock:key <token> NX PX 30000
#+end_src

Release safely:
- verify token ownership
- delete atomically (usually Lua script)

For multi-node lock strategy, Redlock is commonly discussed.

* Key-Value & Caching (Redis + DynamoDB)
** Q100. What is Redis Cluster?
Redis Cluster is Redis’s built-in sharding system.

- Keyspace is split into *16384 hash slots*
- Slots are distributed across primary nodes
- Each primary can have replicas for failover
- Clients are redirected to the node owning the slot for a key

*Why it matters:* lets Redis scale horizontally.

** Q101. Is Redis single-threaded?
Mostly yes for command execution:
- Core command processing is effectively single-threaded (simple atomic ops, no lock complexity)
- Modern Redis uses extra threads for I/O and background tasks

So, command logic is serialized, but not every internal task is single-threaded.

** Q102. What is DynamoDB?
DynamoDB is AWS’s fully managed NoSQL service (key-value + document).

Key traits:
- low-latency reads/writes (single-digit ms typical)
- automatic scaling
- serverless operations
- configurable capacity/throughput modes

** Q103. Partition key and sort key in DynamoDB
- *Partition key* determines where data is stored
- If only partition key exists, it must uniquely identify an item
- With a *sort key*, uniqueness is based on (partition key + sort key)
- Items sharing partition key are ordered by sort key

This enables efficient range queries inside a partition.

** Q104. Query vs Scan in DynamoDB
- *Query*: efficient, targeted (uses partition key and optional sort-key conditions/indexes)
- *Scan*: reads entire table/index then filters (expensive, slower)

*Rule:* avoid Scan in hot production paths.

** Q105. GSIs vs LSIs in DynamoDB
- *GSI (Global Secondary Index)*:
  - can use different partition/sort keys
  - separate throughput
  - eventually consistent reads
- *LSI (Local Secondary Index)*:
  - same partition key as base table
  - different sort key
  - supports strong consistency
  - must be defined at table creation

** Q106. DynamoDB consistency options
- *Eventually consistent reads* (default, cheaper)
- *Strongly consistent reads* (optional on table/LSI, higher cost/latency)
- GSIs provide eventual consistency
- Writes are durably stored across multiple AZs

* Wide-Column Stores (Cassandra-focused)
** Q107. What is Apache Cassandra?
Cassandra is a distributed, masterless wide-column database designed for:
- high availability
- linear horizontal scaling
- high write throughput
- no single point of failure

Strong choice for always-on, write-heavy, multi-DC workloads.

** Q108. Cassandra data model
- *Keyspace* contains tables
- Table primary key = partition key + optional clustering columns
- Partition key controls placement
- Clustering columns control ordering in partition
- Rows are sparse (not all rows need same columns populated)
  
** Q109. What is CQL?
CQL (Cassandra Query Language) is SQL-like syntax for Cassandra.

Looks familiar, but:
- no arbitrary joins
- limited ad-hoc aggregations
- efficient queries must align with partition key design

** Q110. Why model around queries in Cassandra?
Cassandra reads efficiently by partition key access.
Therefore, schema is designed from access patterns first.

Common approach:
- one table per major query pattern
- deliberate duplication for speed/predictability
  
** Q111. How Cassandra distributes data
- partition key is hashed onto a token ring
- nodes own token ranges
- vnodes (virtual nodes) split ownership into many smaller ranges for better balancing

** Q112. Replication factor (RF)
RF = number of copies of each partition (e.g., RF=3).

Higher RF:
- improves availability/durability
- increases storage/network cost

Configured per keyspace.

** Q113. Tunable consistency in Cassandra
Per operation, choose consistency level:
=ONE=, =QUORUM=, =LOCAL_QUORUM=, =ALL=, etc.

If read CL + write CL > RF, you can get strong read-after-write behavior.
Lower CL improves latency/availability but may read stale data.

** Q114. What does QUORUM mean?
QUORUM = majority replicas:
=floor(RF/2) + 1=

- =QUORUM=: across all relevant replicas
- =LOCAL_QUORUM=: majority in local DC (common for multi-DC latency control)

** Q115. Cassandra write path
Write flow:
1. append to commit log (durability)
2. update memtable (in memory)
3. flush memtable to SSTables (immutable files on disk)

No read-before-write needed -> fast write path.

** Q116. What is compaction?
Compaction merges SSTables in background to:
- remove obsolete versions/tombstones
- reduce read overhead
- reclaim disk space

Common strategies:
- STCS
- LCS
- TWCS (good for time-series windows)

** Q117. What is a tombstone?
A tombstone is a delete marker (not immediate physical deletion).
Actual data is removed during compaction after =gc_grace_seconds=.
Too many tombstones hurt read performance.

** Q118. Why deletes can be problematic in Cassandra
Frequent deletes/updates create many tombstones.
Until compaction cleans them, range reads become slower and can hit tombstone-heavy query issues.

** Q119. What is read repair?
On reads, Cassandra can detect replica differences and repair stale replicas by writing newer values back.
Purpose: improve replica convergence over time.

** Q120. What is hinted handoff?
If a target replica is down during write:
- coordinator stores a hint
- replays hint when replica returns

This improves availability and eventual convergence.

** Q121. Last-write-wins (LWW) in Cassandra
Conflicts are resolved by timestamp; latest timestamp wins.

Risk:
- clock skew or concurrent updates can cause unexpected data loss.

** Q122. HBase vs Cassandra (high-level)
- *HBase*: CP-leaning, master-based, HDFS ecosystem, strong consistency
- *Cassandra*: AP-leaning, masterless, tunable consistency, multi-DC resilience, write-heavy strength

** Q123. Column family vs super column
- *Column family*: main logical column grouping
- *Super columns*: old nested model, now legacy/deprecated in favor of modern schema patterns (composite keys/collections)

** Q124. Cassandra collection types
Supports:
- =set=
- =list=
- =map=
- UDTs (user-defined types)

Use collections for small bounded data.
Large/unbounded collections should be modeled as separate rows.

** Q125. Materialized views in Cassandra
A materialized view auto-maintains another table shape from base table with different primary key.

Benefit: alternate query path.
Caution: known consistency/operational complexities; use carefully.

** Q126. Limits of Cassandra secondary indexes
Secondary indexes can cause cross-node scatter-gather for many query types and may perform poorly at scale.

Prefer:
- query-specific denormalized tables
- modern indexing features (e.g., SAI where applicable)

* Graph Databases (Neo4j-focused)

** Q127. What is the property graph model?
Data is represented as:
- *nodes* (entities)
- *relationships* (directed edges)
- properties on both
- labels/types for categorization

Relationships are first-class and queryable directly.

** Q128. What is Cypher?
Cypher is Neo4j’s declarative graph query language using pattern syntax.

#+begin_src cypher
MATCH (a:Person)-[:FRIEND]->(b:Person)
WHERE a.name = 'Ada'
RETURN b.name;
#+end_src

** Q129. Create nodes and relationships in Cypher
#+begin_src cypher
CREATE (a:Person {name: 'Ada'})
CREATE (b:Person {name: 'Alan'})
CREATE (a)-[:FRIEND {since: 2020}]->(b);
#+end_src

** Q130. Best use cases for graph databases
Use when relationships are central:
- social networks
- recommendation engines
- fraud/link analysis
- IT/network topology
- knowledge graphs
- pathfinding problems

** Q131. Why graphs outperform relational DBs on deep traversals
Graph DBs use direct edge traversal (index-free adjacency), so each hop is cheap.
Relational systems typically need repeated joins for multi-hop paths, which gets expensive quickly.

** Q132. What is index-free adjacency?
Nodes maintain direct references to connected nodes/edges.
Traversal follows pointers instead of repeated global index lookups.

This is a core graph-performance advantage for relationship-heavy workloads.

** Q133. Shortest path in Cypher
#+begin_src cypher
MATCH p = shortestPath(
  (a:Person {name:'Ada'})-[:KNOWS*]-(b:Person {name:'Grace'})
)
RETURN p;
#+end_src

** Q134. Variable-length traversal
Patterns like =[:KNOWS*1..3]= match 1 to 3 hops.
Great for “friends of friends” style queries.

** Q135. Property graph vs RDF/triple store
- *Property graph*: properties on nodes/edges, labels, imperative-like traversal style
- *RDF*: subject-predicate-object triples with URIs, queried via SPARQL, strong for linked/semantic web data

** Q136. What is =MERGE= in Cypher?
=MERGE= means “match if exists, otherwise create.”
Useful for idempotent upserts and duplicate prevention.

** Q137. How graph DBs scale
Graph partitioning is harder because traversals can cross boundaries.

Common strategies:
- read replicas
- caching
- domain-based partitioning (community/locality)
- distributed graph engines for large-scale workloads

** Q138. Graph recommendation use case
Example: “people who bought X also bought Y” or “users with similar interaction neighborhoods.”

Graph traversal naturally models these relationship patterns.

** Q139. ACID in Neo4j
Neo4j provides ACID transactions (in core/clustered modes), which is valuable when relationship integrity and transactional correctness matter.

** Q140. Graph projection / GDS
Neo4j Graph Data Science (GDS) runs algorithms on in-memory graph projections:
- PageRank
- community detection
- centrality
- similarity

Used for analytics, ranking, and ML feature generation.

** Q141. When NOT to use graph DB
Avoid graph DB when workload is mainly:
- simple key-value lookups
- flat bulk aggregations
- extreme write-heavy non-relationship workloads

Other models are often simpler and cheaper there.

* Consistency, Distribution & Internals
** Q142. Quorum-based consistency
Reads/writes require acknowledgments from a majority of replicas.
If read/write quorums overlap, latest writes are visible.

** Q143. R + W > N formula
Where:
- N = replica count
- W = write acknowledgments required
- R = replicas consulted on read

If =R + W > N=, read and write sets overlap -> stronger consistency.

** Q144. What is a vector clock?
A versioning mechanism (counter per node/actor) to track causality between updates.
Helps detect concurrent conflicts instead of blindly overwriting.

** Q145. What is a conflict and how to resolve?
Conflict = concurrent divergent updates.

Common resolution strategies:
- last-write-wins
- vector-clock sibling merge
- app-defined merge logic
- CRDT convergence

** Q146. What are CRDTs?
CRDTs are data types designed so concurrent updates merge deterministically without coordination.

Benefits:
- eventual convergence
- fewer write conflicts in distributed active-active setups

** Q147. Influence of the Dynamo paper (2007)
Key ideas popularized:
- consistent hashing
- vector clocks
- quorums
- hinted handoff
- gossip membership

These concepts influenced Cassandra, Riak, DynamoDB-style systems.

** Q148. Consistent hashing
Maps nodes and keys onto a ring.
Adding/removing a node remaps only a fraction of keys (roughly 1/N), reducing rebalance cost.

** Q149. What are virtual nodes (vnodes)?
Each physical node owns many small token ranges instead of one big range.
Improves load balance and smoother scaling/recovery.

** Q150. Gossip protocol
Nodes periodically exchange cluster-state info with peers.
State spreads quickly without a central coordinator.

Used for membership and failure detection.

** Q151. Merkle tree and role
Merkle tree summarizes data ranges by hashes.
Replicas compare hashes first to find differences efficiently, then sync only mismatched ranges.

** Q152. Anti-entropy repair
Background process that reconciles replica differences (often using Merkle-tree comparison).
Complements hinted handoff and read repair.

** Q153. Read-your-own-writes consistency
Session guarantee: a client sees its own previous writes.
Often achieved via sticky routing/session affinity.

** Q154. Monotonic reads
Once a client has seen a newer value, it should never read an older one later.

** Q155. Causal consistency
Preserves cause-effect order of related operations globally.
Concurrent unrelated operations may still be seen in different orders.

** Q156. PACELC theorem
Extends CAP:
- If *Partition* -> trade *Availability vs Consistency* (A/C)
- Else (normal case) -> trade *Latency vs Consistency* (L/C)

Captures real-world latency cost of stronger consistency.

** Q157. WAL / commit log
Write-ahead log stores changes durably before applying to main structures.
Enables crash recovery and durability.

** Q158. LSM tree
Write-optimized storage model:
1) write to memtable
2) flush to immutable SSTables
3) background compaction merges SSTables

Used by Cassandra, HBase, RocksDB, LevelDB.

** Q159. LSM tree vs B-tree
- *B-tree*: in-place updates, strong random-read behavior, random-write overhead
- *LSM*: append/merge write path, very write-friendly, needs compaction and can increase read complexity

** Q160. Bloom filter in NoSQL
Probabilistic “maybe present?” check:
- no false negatives
- possible false positives

Helps skip disk files that definitely do not contain a key.

** Q161. Write amplification
Physical bytes written / logical bytes written.
Compaction can rewrite data multiple times, increasing this ratio.

** Q162. Read amplification
Extra reads needed to answer one logical query (e.g., checking multiple SSTables/indexes).
Compaction and bloom filters help reduce it.

** Q163. Space amplification
Extra disk used beyond logical dataset size due to old versions, tombstones, compaction overlap.

** Q164. Sync vs async replication
- *Synchronous replication*: wait for replica ack before success -> stronger consistency, higher latency
- *Asynchronous replication*: ack early, replicate later -> lower latency, risk window for data loss

** Q165. Split-brain scenario
During partition, multiple nodes/groups act like primary/leader and accept conflicting writes.

Prevention:
- majority quorum
- fencing tokens
- robust leader election

** Q166. Leader election
Protocol to pick one leader/coordinator in distributed cluster (Raft/Paxos style).
Used in systems like MongoDB replica sets and other distributed controllers.

** Q167. Raft vs Paxos
Both are consensus algorithms.
- *Raft*: simpler mental model, leader-centric log replication
- *Paxos*: foundational/older, more complex to reason about/implement

* Operations, Security & Advanced
** Q168. Backup strategies for NoSQL
Common approaches:
- snapshots
- logical dumps
- continuous backups / PITR
- managed-service backups

Most important: *test restore regularly*.

** Q169. =mongodump= vs filesystem snapshot
- =mongodump/mongorestore=: logical, portable, slower/larger
- filesystem/volume snapshot: fast for large data, infra-specific, good PIT consistency with proper setup

** Q170. Monitoring a NoSQL cluster
Track:
- p95/p99 latency
- throughput
- error rate/timeouts
- replication lag
- disk/memory/CPU
- compaction/GC
- hot partitions
- node health

Use Prometheus/Grafana, native tools, or cloud dashboards.

** Q171. Replication lag
Delay between primary write and replica visibility.

Why it matters:
- stale reads from replicas
- larger failover data-loss window in async setups

** Q172. Securing NoSQL systems
- enable auth + RBAC
- TLS in transit
- encryption at rest
- network isolation (VPC/firewalls/private endpoints)
- least privilege credentials
- audit logging
- disable insecure defaults/open binds

** Q173. NoSQL injection risk
If user input is inserted unsafely into query objects/operators, attackers may manipulate query logic.

Mitigate via:
- strict input validation
- parameterized query building
- avoiding dangerous features (=where/eval-style execution)

** Q174. Large-scale migration in NoSQL
Safe pattern:
1) dual-write old/new model
2) backfill history in batches
3) verify/reconcile
4) switch reads
5) retire old path

Use CDC/change streams + schema versions for low downtime.

** Q175. What is CDC?
Change Data Capture streams row/document changes to downstream systems.
Use cases:
- search indexing
- cache sync
- analytics/lake ingestion
- event-driven services

** Q176. Integrating NoSQL with search engines
Use CDC to push DB changes to Elasticsearch/OpenSearch.
Primary DB stays source of truth; search index is eventually consistent projection.

** Q177. Lambda vs Kappa architecture
- *Lambda*: separate batch + real-time layers
- *Kappa*: one streaming pipeline, replay stream when needed
  
Both often use NoSQL + stream infra in serving paths.

** Q178. Data locality in distributed NoSQL
Keep data near where it is accessed (partition/node/DC/region) to reduce network hops and latency.

** Q179. Multi-region NoSQL deployments
Patterns:
- active-passive
- active-active

Active-active improves availability but needs conflict strategy (LWW, merge rules, CRDT-like approaches) and accepts cross-region consistency lag.

** Q180. OLTP vs OLAP in NoSQL context
- NoSQL primary stores are usually optimized for OLTP (fast transactional access paths)
- OLAP is often offloaded to warehouses/columnar engines/search systems via ETL/CDC

** Q181. Idempotency for NoSQL writes
Idempotent operation gives same result even if retried.
Essential in distributed retries/at-least-once delivery to prevent duplicates/double updates.

** Q182. Conditional writes / optimistic concurrency
Writes succeed only if condition matches expected state:
- version check
- existence/non-existence
- explicit condition expressions

Prevents lost updates without pessimistic locks.

** Q183. Counters at scale
Use atomic increments (e.g., Redis =INCR=).
For very high write rates, shard counters across multiple keys and sum at read time to avoid hot keys.

** Q184. Dual-write problem
Writing same event/state to two systems independently can leave them inconsistent on partial failure.

Prefer:
- outbox pattern
- CDC/event relay
instead of naive dual writes.

** Q185. Outbox pattern
Write business data + outbox event in one atomic transaction.
A relay publishes outbox events later.

Guarantee: state change and emitted event remain consistent.

** Q186. Application impact of eventual consistency
Apps should:
- tolerate stale reads
- use idempotent operations
- handle conflicts/retries
- avoid strict read-after-write assumptions unless session guarantees exist
- show pending/intermediate status in UX when needed

** Q187. Choosing document vs key-value vs wide-column vs graph
Choose by dominant access pattern:
- rich nested records/flexible docs -> document
- ultra-fast key lookup/cache -> key-value
- massive write-heavy predictable queries/time-series -> wide-column
- deep relationship traversal -> graph

** Q188. Polystore / multi-model database
One platform supports multiple models in same engine (document + key-value + graph-like features, etc.).
Benefit: lower ops overhead vs running many specialized databases.

** Q189. Common NoSQL anti-patterns
- modeling like normalized SQL everywhere
- poor partition key (hotspots)
- unbounded arrays/partitions
- heavy scans instead of query-driven design
- ignoring consistency semantics
- no migration/version strategy

** Q190. Testing and benchmarking NoSQL
Use realistic workload shape:
- read/write mix
- key distribution
- concurrency
- failure/partition scenarios
- percentile latency (p95/p99), not just averages

YCSB is a common baseline benchmark tool.

** Q191. Future: NoSQL and NewSQL convergence
Boundaries are blurring:
- NewSQL adds distributed SQL + strong transactions at scale
- NoSQL adds transactions, richer querying, and stronger consistency options

Direction: practical convergence toward distributed data platforms with flexible consistency/performance trade-offs.
